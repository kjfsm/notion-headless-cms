# v0.2 → v0.3 移行ガイド

v0.3.0 は実質的な v1-RC であり、リリース前に破壊的変更を一度に入れた。
以降はセマンティックバージョニングを厳格に守る。

## 主な破壊的変更

1. **`createNodeCMS` / `createCloudflareCMS` 廃止** — `createCMS` 一本に統一
2. **`adapter-node` / `adapter-cloudflare` パッケージ削除** — `nodePreset` (core) / `cloudflarePreset` (cache-r2) に統合
3. **`InvalidateScope` の簡素化** — `{ collection, slug? }` 形式に一本化 (`{ slug }` / `{ tag }` 廃止)
4. **NHC プレフィクス → CMS プレフィクス** — `NHCConfig` → `CMSConfig`, `nhcDataSources` → `cmsDataSources` 等
5. **CLI / core の生 Error → CMSError** — `cli/*` 名前空間を新設
6. **`publishConfig.exports` 重複削除** — 発行挙動に影響なし
7. **`cache-next` の invalidate を規約タグへ** — `nhc:col:<name>` / `nhc:col:<name>:slug:<slug>`

## ランタイムファクトリの移行

### Node.js

```ts
// v0.2 (Before)
import { createNodeCMS } from "@notion-headless-cms/adapter-node";
const cms = createNodeCMS({
  dataSources: nhcDataSources,
  cache: { document: "memory", image: "memory", ttlMs: 5 * 60_000 },
});
```

```ts
// v0.3 (After)
import { createCMS, nodePreset } from "@notion-headless-cms/core";
const cms = createCMS({
  ...nodePreset({ ttlMs: 5 * 60_000 }),
  dataSources: cmsDataSources,
});
```

### Cloudflare Workers

```ts
// v0.2 (Before)
import { createCloudflareCMS } from "@notion-headless-cms/adapter-cloudflare";
const cms = createCloudflareCMS({
  dataSources: nhcDataSources,
  env,
  ttlMs: 5 * 60_000,
});
```

```ts
// v0.3 (After)
import { createCMS } from "@notion-headless-cms/core";
import { cloudflarePreset } from "@notion-headless-cms/cache-r2";
const cms = createCMS({
  ...cloudflarePreset({ env, ttlMs: 5 * 60_000 }),
  dataSources: cmsDataSources,
});
```

推奨 binding 名は `DOC_CACHE` (KV) と `IMG_BUCKET` (R2)。
旧 `CACHE_KV` / `CACHE_BUCKET` もフォールバックとして認識する。
カスタマイズする場合:

```ts
cloudflarePreset({ env, bindings: { docCache: "MY_KV", imgBucket: "MY_R2" } })
```

## NHC → CMS 置換コマンド

```bash
# package.json から adapter-* 依存を外す (手動)
# import と変数名の一括置換
grep -rl "nhcDataSources" src/ examples/ 2>/dev/null | xargs sed -i 's/nhcDataSources/cmsDataSources/g'
grep -rl "NHCConfig"  src/ examples/ 2>/dev/null | xargs sed -i 's/NHCConfig/CMSConfig/g'
grep -rl "NHCSchema"  src/ examples/ 2>/dev/null | xargs sed -i 's/NHCSchema/CMSSchema/g'
grep -rl "NHCDataSources" src/ examples/ 2>/dev/null | xargs sed -i 's/NHCDataSources/CMSDataSources/g'

# 生成済み nhc-schema.ts を再生成
pnpm nhc generate
```

`nhc.config.ts` / `nhc` CLI コマンド / `nhc-schema.ts` ファイル名は維持される。

## `InvalidateScope` の移行

```ts
// Before
cms.$revalidate({ slug: "my-post" });
cms.$revalidate({ tag: "posts" });

// After — slug だけでは呼べない。collection を必ず指定する
cms.$revalidate({ collection: "posts", slug: "my-post" });
cms.$revalidate({ collection: "posts" });
cms.$revalidate("all");
```

コレクション単位の API (`cms.posts.revalidate({ slug })`) はそのまま使える。

## CMSError の移行

CLI が throw するエラーが生の `Error` から `CMSError` に変わった。
`instanceof Error` は引き続き true だが、より詳細な判定をしたい場合:

```ts
import { isCMSErrorInNamespace } from "@notion-headless-cms/core";

try {
  await runGenerate(opts);
} catch (err) {
  if (isCMSErrorInNamespace(err, "cli/")) {
    console.error(`[${err.code}] ${err.message}`);
  } else {
    throw err;
  }
}
```

## `cache-next` の tag 規約

`invalidate` / `$revalidate` に `{ collection }` / `{ collection, slug }` を
渡すと、内部で以下のタグを `revalidateTag` する:

- `{ collection: "posts" }` → `nhc:col:posts`
- `{ collection: "posts", slug: "abc" }` → `nhc:col:posts` と `nhc:col:posts:slug:abc`
- `"all"` → `nextCache({ tags })` で指定したユーザー定義タグ全て

Next.js の `fetch` / `unstable_cache` 側で規約タグを付ければ連携できる。
